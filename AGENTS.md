# AGENTS.md — 이 프로젝트에서 항상 먼저 읽는 안내서

이 파일은 AI 코딩 도우미가 이 프로젝트에서 일할 때 매번 참고하는 기준이다.
작성자는 코딩을 처음 배우는 학생이므로, 설명은 쉽게 하고 용어는 짧게 풀어 써야 한다.

## 0. 가장 중요한 규칙

- 어려운 전문 용어가 나오면 괄호로 짧게 뜻을 붙인다.
- 한 번에 많은 내용을 몰아주지 말고 작은 단계로 나눠 설명한다.
- 비유를 곁들이면 더 좋다.
- 명령만 하지 말고, 왜 그렇게 하는지도 한 줄 덧붙인다.
- 개인정보(학생 실명, 학번, 출결, 상벌점)는 민감하니 함부로 보여주거나 바꾸지 않는다.

이유: 초보자는 용어와 흐름이 한꺼번에 나오면 이해하기 어렵기 때문에, 읽는 부담을 먼저 줄이기 위해서다.

## 1. 이 프로젝트가 해결하려는 것

- 학교 생활에 필요한 정보가 여러 곳에 흩어져 있어 찾기 불편한 문제를 줄인다.
- 출결관리와 상벌점관리처럼 학교 생활에 중요한 기능을 한 앱 안에 모은다.
- 학생이 매일 쓰기 편하고, 교사가 관리하기도 쉬운 앱을 만든다.

이유: 이 프로젝트의 방향을 잊지 않도록, 무엇을 해결하려는지 먼저 고정해 두기 위해서다.

## 2. 최종 기술 스택과 고른 이유

> 스택(Stack) = 앱을 만들 때 쓰는 기술들의 묶음.

| 역할 | 고른 기술 | 왜 이걸 쓰는지 | 초보가 배우기 쉬운지 | 무료인지 | 다른 후보 |
|---|---|---|---|---|---|
| 앱 화면 | Flutter | 화면을 한 번에 만들기 쉽고, iOS와 Android를 같이 볼 수 있다. | 쉬운 편이다. 자료와 예제가 많다. | 무료다. | React Native, Kotlin, Swift |
| 화면 뒤 처리 | Azure Functions | 출결 기록, 상벌점 처리, 검증 같은 일을 서버에서 담당한다. | 중간 정도다. 개념은 쉽지 않지만 관리가 편하다. | 무료 구간이 있다. | Node.js + Express, NestJS, Django |
| 로그인과 권한 | Azure AD B2C 또는 Entra 계열 인증 | 학생과 관리자를 구분하고, 누가 무엇을 볼 수 있는지 나눈다. | 중간 정도다. 인증은 꼭 배우면 좋지만 처음엔 조금 어렵다. | 무료 구간이 있다. | 직접 로그인 서버, 다른 인증 서비스 |
| 데이터 저장 | Azure Cosmos DB 또는 Azure Database 계열 | 출결, 상벌점, 공지 같은 정보를 오래 저장한다. | 중간 정도다. 데이터 구조를 배워야 한다. | 무료 구간이 제한적으로 있다. | PostgreSQL, MySQL, Firestore |
| 파일 저장 | Azure Blob Storage | 사진, 첨부파일, 자료를 보관한다. | 쉬운 편이다. 파일을 넣고 꺼내는 느낌이라 이해하기 쉽다. | 무료 구간이 있다. | Cloud Storage, S3 |
| 알림 | Azure Notification Hubs 또는 앱 알림 연동 | 공지, 출결 관련 알림을 사용자에게 보낸다. | 중간 정도다. 푸시 알림은 처음에 낯설 수 있다. | 무료 구간이 있거나 제한적이다. | Firebase Cloud Messaging, OneSignal |
| 실시간 전달 | Socket.io 또는 Azure 실시간 연결 방식 | 채팅, 즉시 반영이 필요한 화면에 쓴다. | 중간 정도다. 연결 관리가 필요하다. | 방식에 따라 다르다. | Firebase 실시간 기능, SignalR |
| 외부 급식 연동 | NEIS 오픈 API | 급식 정보를 공식 데이터로 가져온다. | 쉬운 편이다. 요청과 응답 구조가 단순하다. | 대체로 무료다. | 수동 입력, 자체 데이터 입력 |
| 기기 연결 | Flutter용 BLE 패키지 | 가까운 기기와 연결해 등교 기록 같은 기능을 처리한다. | 중간에서 어려운 편이다. 블루투스 특성을 배워야 한다. | 무료다. | flutter_reactive_ble, 다른 BLE 패키지 |

이유: 초보자에게는 "많이 쓰이고 자료가 많은 기술"이 배우기 쉽기 때문에, 그 기준으로 칸을 나눠 적었다.

## 3. 만들 핵심 기능

- 학생 인증과 상벌점 현황을 제공한다.
- 전체 익명 채팅과 학년별 익명 채팅을 제공한다.
- 학교 공지, 이벤트, 온라인 가정통신문, 공지사항을 제공한다.
- 시간표, 급식, D-day, 행사 알림, 동아리 홍보 및 신청, 호림재 예약을 제공한다.
- 마이페이지, 익명 프로필, 레벨, 뱃지, 감정버튼, 공부시간 카운트를 제공한다.
- 시간표, D-day, 급식 위젯 설정을 제공한다.
- BLE 기반 등교 시간 기록과 교문 앞 태블릿 거치 인증을 제공한다.

이유: 처음부터 전부 구현하려고 하면 범위가 너무 커지므로, PRD에 적힌 실제 요구사항만 핵심 기능으로 모아 두기 위해서다.

## 4. 이번엔 만들지 않을 것

- 다른 학교와의 데이터 공유나 학교 간 커뮤니티
- 결제나 비용 정산 기능
- 성적, 내신 산출, 교육과정 관리
- 교사용 행정 관리 도구의 별도 확장 기능

이유: 범위를 처음부터 너무 넓히면 완성하기 어려워서, 지금은 학교 앱의 핵심 기능에만 집중하기 위해서다.

## 5. 작업할 때 지켜야 할 방식

- 코드를 바꾸기 전에 먼저 PRD.md와 이 파일을 확인한다.
- 아직 정해지지 않은 요구사항은 임의로 채우지 말고 확인 질문을 한다.
- 새 용어를 쓸 때는 반드시 짧은 설명을 붙인다.
- 기능을 나눠서 조금씩 만든다.

이유: 초보자에게는 한 번에 많은 변경보다, 작은 변경과 확인이 훨씬 안전하기 때문이다.

## 6. 이 파일을 이렇게 나눈 이유

- 1번은 왜 이 앱이 필요한지부터 잡아 주기 위해 넣었다.
- 2번은 어떤 기술을 쓰는지와 왜 그걸 고르는지 바로 보이게 하려고 넣었다.
- 3번은 지금 만들 범위를 헷갈리지 않게 하려고 넣었다.
- 4번은 처음부터 건드리지 않을 부분을 분명히 하려고 넣었다.
- 5번은 앞으로 작업할 때의 공통 규칙을 고정하려고 넣었다.

이유: 큰 방향, 기술 선택, 기능 범위, 제외 범위, 작업 규칙을 순서대로 보면 초보자도 문서를 빨리 이해할 수 있기 때문이다.
